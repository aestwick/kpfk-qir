KPFK / Pacifica — Confessor & Archive API Reference

v1 draftread-onlyGET only

Single-endpoint HTTP API for show schedules, "now playing" data, playlists, fund-drive config, and on-demand audio. Powers the broadcast side of the Pacifica network and is the upstream that Beacon (Otis) reads from to link donations to whatever show is on air. Everything is dispatched off one query parameter: ?req=<endpoint>.

This page is both the documentation and a working test harness. Shapes marked ✓ verified live 2026-06-17 were confirmed against the deployed confessor.kpfk.org server (all 26 endpoints). The remaining few are labeled from source. See CAVEATS.md for the full evidence ledger.

Overview

Two PHP scripts implement the same API, one per generation. Each station runs its own host; the path and the ?req= contract are identical across stations.

ConceptValue
Current base URL (KPFK)https://confessor.kpfk.org/_nu_do_api.php
Legacy base URLhttp://confessor.kpft.org/_do_api.php (KPFT host in old config)
Other stationsconfessor.<station>.org — e.g. kpft.org, wpfw.org. Same paths.
MethodGET only. No auth. No write operations.
Dispatch?req=<name> selects the endpoint. Unknown/empty req → empty body (or literal bad on nu).
Default responsebase64-encoded PHP serialize() — not JSON. See below.
JSON responseAppend &json=1 to any request. Always do this from non-PHP clients.
Why this exists for Beacon. Beacon never writes here. It reads three things: which show is on air now (for live-show donation attribution), the show list for dropdowns, and archive MP3 links. Beacon's programs.slug maps to Confessor's sh_altid — that slug is the join key between the two systems.

Output encoding — read this first ⚠

Source vs. production drift (verified live 2026-06-17). The PHP in this repo defaults to base64_encode(serialize($array)) and switches to JSON on &json=1. But the deployed confessor.kpfk.org server was observed returning plain JSON even without &json=1 (confirmed on getnow and special with no flag). Production is running a newer build than this export. Takeaway: always send &json=1 anyway — it's correct against both the old serialize-default and the new JSON-default, so your client never has to care which build it hits.

So depending on which build answers you:

BuildNo json=1With &json=1
This repo's sourcebase64(serialize(...))JSON
Deployed KPFK (observed)JSONJSON
Gotcha: the JSON path uses JSON_NUMERIC_CHECK, so numeric strings come back as numbers. IDs and timestamps may be JSON numbers, not strings — don't assume string types when parsing.

Decoding the serialized form (only needed against the old build)

// PHP (the intended path)
$buf = file_get_contents("https://confessor.kpfk.org/_nu_do_api.php?req=getday&day=2");
$ary = unserialize(base64_decode($buf));

// JSON path — preferred everywhere else
curl -s "https://confessor.kpfk.org/_nu_do_api.php?req=getday&day=2&json=1"
// JavaScript (browser / Node) — always use json=1
const r = await fetch(URL + "?req=getnow&json=1");
const data = await r.json();
Header caveat: several endpoints emit PHP notices/warnings before the body when something is off (see Known issues). That text is prepended to the JSON, so a strict JSON.parse can fail. Defensive clients should trim to the first {/[.

Two engines: _do_api.php (old) vs _nu_do_api.php (nu)

The nu engine is the current, more comprehensive one and is what the KPFK base URL points at. The old engine is a strict subset still deployed at some hosts. Document and build against nu; treat old as deprecated/fallback.

Each wrapper function takes a trailing $json_flag argument. When set it appends &json=1 and returns the raw JSON string instead of an unserialized array.
